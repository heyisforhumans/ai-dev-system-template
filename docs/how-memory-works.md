# How Memory Works — The 3-Tier System

> Your AI reads these files to remember your project. The better they're maintained, the smarter the AI gets over time.

---

## The Problem

AI coding assistants have no memory between sessions by default. Every conversation starts fresh. Without a memory system, you spend the first 10 minutes of every session re-explaining what you built, what decisions you made, and where things are.

This system solves that with three tiers of memory — each serving a different purpose.

---

## The 3 Tiers

```
┌─────────────────────────────────────────────────────────┐
│  TIER 1 — Always Loaded (Automatic)                     │
│  Broad knowledge your AI loads automatically            │
│  Antigravity: Knowledge Items (KI) system               │
│  Other tools: paste BOOTSTRAP.md at session start       │
├─────────────────────────────────────────────────────────┤
│  TIER 2 — Session Context (Per-App)                     │
│  .agents/memory/[app-name].md                           │
│  Current state, decisions, gotchas, open questions      │
│  Updated at the end of every session via /save-memory   │
├─────────────────────────────────────────────────────────┤
│  TIER 3 — Component Docs (Deep)                         │
│  [app]/memory/[feature]_component_memory.md             │
│  AI-generated after completing major features           │
│  10 sections: architecture, API, gotchas, extensions    │
└─────────────────────────────────────────────────────────┘
```

---

## Tier 1 — Always Loaded

**What goes here:** Broad facts about your workspace. What apps exist, what the stack is, what the overall architecture looks like.

**How it's maintained:**
- **Antigravity users**: the KI (Knowledge Item) system automatically distills knowledge after sessions. You don't need to do anything.
- **Other AI tools**: copy the contents of `BOOTSTRAP.md` into your system prompt or paste it at the start of each session.

**You don't manage Tier 1 directly.** It's automatic.

---

## Tier 2 — Session Context

**Location:** `.agents/memory/[app-name].md`

**What goes here:**
- What was worked on and when
- Architecture decisions and why they were made
- Current state of the app (working? deployed? broken? why?)
- Gotchas — things that bit you and how to avoid them
- Open questions that need to be answered
- Component memory index (what Tier 3 files exist)

**How to update:** Run `/save-memory` at the end of every session. The AI writes the summary for you.

**Example entry:**
```markdown
## 2026-03-19 — Notification System
- Added Notification model and NotificationType enum
- Notifications fan out to all users on harvest submit
- Polling interval: 30s in NotificationContext
- Gotcha: if user has no preferences row, API throws 500 — ensure seed creates rows
```

---

## Tier 3 — Component Docs

**Location:** `[app]/memory/[feature]_component_memory.md`

**What goes here:** Deep technical documentation for a specific feature, generated by the AI using `memory_model.py`. Includes:
1. Overview — what it does and why
2. Architecture — files, data flow, key functions
3. API contract — endpoints, request/response shapes
4. Database schema — tables, fields, relationships
5. Frontend integration — which components use it
6. Business logic — the embedded rules
7. Key design decisions — why things are the way they are
8. Gotchas — non-obvious issues and fixes
9. Extension guide — how to add features
10. Testing checklist — manual verification steps

**How to generate:** Run `/memory-model-component` after finishing a major feature.

**Who reads it:** Primarily the AI, when it's about to work on that specific feature. Also useful for human developers onboarding to the codebase.

---

## The Decision Guide

| What kind of fact is it? | Write to |
|---|---|
| "What does this workspace contain?" | Tier 1 (auto) |
| "What is the current state of [app]?" | Tier 2 — `.agents/memory/[app].md` |
| "How do I extend [specific feature]?" | Tier 3 — `[app]/memory/[feature]_component_memory.md` |
| "What are the global infra facts?" | `.agents/global_context.md` |
| "What are the AI system conventions?" | `.agents/ai-dev-system.md` |

---

## The Lifecycle

```
Session starts  → /kickoff reads BOOTSTRAP + Tier 2 files
Work happens    → AI uses Tier 3 docs when working on specific features
Feature done    → /memory-model-component generates new Tier 3 doc
Session ends    → /save-memory writes to Tier 2 files + git commit
Next session    → AI picks up exactly where you left off
```

The longer you use the system, the richer the memory gets — and the less time you spend re-explaining things.
